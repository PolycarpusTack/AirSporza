The Core Challenge
In football, a match has a fixed kickoff time and a predictable duration (90 ± 15 min). A broadcaster can build a rigid schedule around it. In tennis, neither is true. Matches are scheduled in order on a court — each one starts when the previous one ends. Duration ranges from 45 minutes (a bagel-bagel) to 5+ hours (Isner–Mahut). The broadcast model must be completely reimagined.
What breaks from the football model
No scheduled_kickoff_utc — replaced by not_before_utc and estimated_start_utc (dynamically recomputed). No Team entity — replaced by Player with ranking and seed. No Round as a time window — Round is a draw stage (R32, QF, SF, F). No Venue as timezone source — tournament venue is fixed, all matches share one timezone. No fixed broadcast slot — all slots are floating windows with estimated ranges.
Key architectural decisions
Court becomes the primary scheduling unit (replaces fixture time as the organising axis). OrderOfPlay is a first-class entity — the official daily schedule published each morning. BroadcastSlot is always floating — defined by court_id + match position, not a fixed time. A dedicated CascadeEngine continuously recomputes estimated start times as earlier matches complete or run long.

The cascade problem — why tennis breaks the football broadcast model
In football, a delay to one match doesn't affect any other. In tennis, every match on a court is dependent on every earlier match that day. If Match 1 runs 3 hours instead of 1 hour, Match 2 starts 2 hours late, Match 3 starts 4 hours late. The broadcaster planned for a 19:30 prime-time slot. It becomes a 23:30 slot. Scenario D — Rain / Heat suspension (Indian Wells: no roof) Tournament-wide event
Suspension differs fundamentally from a single-match delay
A football rain delay affects one match. At Indian Wells, a rain event suspends all 9 courts simultaneously. The system must handle a TournamentSuspensionEvent — a new entity not needed in football — that propagates a cascade reset across every active match and all queued matches on all courts. A 2-hour rain break at 20:00 UTC pushes every single remaining match of the day by ≥2 hours. For a European broadcaster, this can turn a midnight finish into a 2am finish with zero warning.

Football scheduling model
Fixed kickoff times. Predictable 90±15 min duration. Teams as entities. League round as time window. One match per broadcast slot. Clashes are cross-match conflicts (two matches at 15:00 on different channels). The hard problem is rights management and regional overlays.
Tennis scheduling model
No fixed times — order-based scheduling. Duration 45–300+ min with no ceiling. Individual players as entities. Draw stage as round. Multiple matches per court per day in cascade dependency. The hard problem is duration uncertainty propagation and live slot management.

What the middleware must add for tennis
CascadeEngine — a continuous worker that listens to match completions and recomputes estimated_start_utc for all downstream matches on the same court. Fires slot-changed events to EPG/playout adapters in real time.
OrderOfPlay ingest adapter — daily feed from ATP/WTA/tournament provider. Published ~18:00 local the day before. Creates Match records with order_on_court and not_before constraints.
FloatingSlot model — BroadcastSlot.is_floating = true, anchor_type = FOLLOWS_MATCH. The slot's start time updates continuously as the cascade engine fires. EPG adapters receive range updates, not single timestamps.
CourtSwitch workflow — when a walkover voids a court slot, the system proposes moving a delayed match from another court. Requires scheduler confirmation + audit reason.
TournamentSuspensionEvent — a tournament-scoped status event that sets all active matches to SUSPENDED and all queued matches to delayed by suspension_duration. A single ops action, not per-match.
TBD player binding — player2_id may be null until the previous round match completes. Broadcast slot shows "Winner of [match X]" until binding resolves. Validation blocks publish of any slot with unresolved TBD participants after a configurable deadline.
What stays the same
Despite the differences, the core middleware architecture holds. The outbox pattern, idempotency model, optimistic locking, audit trail, and validation pipeline all apply unchanged. The adapter pattern absorbs the new provider (ATP/WTA live scoring feed). The delta-based draft model still works — schedulers make planning decisions, the cascade engine updates estimates, humans confirm actions. What changes is the timing model within the domain layer, not the infrastructure.
Player entity is evergreen like Team — created once, referenced forever across all tournaments. Ranking updated weekly via provider feed. Seed is tournament-specific (same as how a team's ground/kit varies per season).
Rights model is simpler — tournament deal covers all matches. Per-match rights evaluation (as in football) is replaced by a single tournament-level rights check. The rights pipeline still runs, just with coarser granularity.
Season-start workflow equivalent exists — each year a new Tournament record is created, the draw is seeded, courts are configured. Player records persist. The only new annual task: seed the draw bracket once the draw ceremony happens.
UTC canonical truth still applies — all times stored as UTC, displayed in venue timezone (America/Los_Angeles for Indian Wells). DST handling still matters — US spring-forward is March 8, 2026, during this very tournament.
